## 1. 背景・目的

### 1.1 背景

現状、スタッフに対して **2つのGoogleフォーム**（「ソレイイネ!! 入力フォーム」「MVBe」）を毎回回答してもらっている。運用上の使いづらさがあり、今後は **1つのWebサイト上** で同一の体験として運用したい。

### 1.2 目的

- フォームの入口を一本化し、スタッフの負担と案内コストを下げる。
- スタッフマスタ（氏名・部署・あだ名等）を一元管理し、**登録・退職に応じたリストの自動反映**を行う。
- 回答時に **部署別の一覧** と **名前検索（あだ名含む）** で、対象者を素早く特定できるようにする。
- **ログイン** に基づき、本人のみが **自分の回答内容**（対象者・コメント全文）を確認でき、**今週・今月の回答状況** を把握できるようにする。
- **月間ランキング** により称賛・投票の結果を共有し、**集計** は管理者が利用できるようにする。

## 2. スコープ

### 2.1 対象範囲（本書で定義するもの）

- 統合サイト上での **アンケートフォーム2種** の提供。
- **ログイン機能**（**5.11**）。マイ回答・集計等、本人性・権限が必要な機能の前提とする。
- **マイ回答・履歴**（誰に投票／選出したか、コメント等の全文を **本人のみ** が閲覧。**5.12**）。
- **マイ回答の検索・フィルタ**（期間・フォーム種別。**5.12**）。
- **回答状況の確認**（今週・今月に各フォームを回答したか。**5.13**）。
- **月間ランキング**（過去月の参照。**5.14**）および **ランキング画面の固定ヘルプ**（集計ルールの表示。**5.14**）。
- **集計機能**（**管理者** アカウントに限定して UI および機能を提供。**5.15**）。
- アンケート画面を含む **UI方針のたたき台**（**5.10**。**ダッシュボード型シェル**（左サイドバー・上ヘッダー・中央ワークスペース）、**ログイン画面** を含む。ピクセル単位・最終ビジュアルは **2.2** のとおり詳細設計・デザインで確定する）。
- **スタッフ登録**・**退職** に連動したマスタ更新（上記2アンケートで利用するスタッフリストへの反映／削除）。
- 部署単位のグルーピング表示、および名前検索（あだ名対応）。
- スタッフ属性（氏名、部署、あだ名等）の管理。**当面は Google スプレッドシート** をマスタの正とする（**6章**）。

### 2.2 対象外（本書では詳細を定めないもの）

- 画面レイアウトのピクセル単位の再現、Levelaロゴ等のビジュアルアセットの最終仕様（文言・設問は **5.7〜5.9** で定義）。
- 認証方式の **技術的詳細**（Googleアカウント連携、社内SSO、パスワード等の **採用プロトコル・プロバイダ**）の最終決定（**8章**）。**ログイン機能の提供** および **本人・管理者の区別** は **5.11・5.15** で要件化する。
- **2.3** で確定した範囲を超えるインフラの細部（リージョン指定、監視ツールの個別選定等）。

### 2.3 技術・インフラ方針（確定）

次の方針で実装・運用する。

| 区分 | 内容 |
|------|------|
| フロントエンド | **Next.js**（App Router を想定） |
| ホスティング | **Vercel** |
| マスタの保存先 | **Google スプレッドシート**（1ブック内の専用シート。正のデータは当該シートのみとし、二重管理を避ける） |
| サイト上アンケートの回答保存先 | **同一または別ブックの Google スプレッドシート**（アンケート種別ごとにシートを分ける。**6.3**） |
| ブラウザからスプレッドシートへの直接アクセス | **行わない**。API 等のサーバー側で Google Sheets API を呼び出し、鍵・権限をサーバー環境（Vercel の環境変数等）に保持する。 |
| Google フォーム（登録・退職） | 引き続き利用。**マスタ用スプレッドシート** を更新する経路を設計・実装で一本化する（**5.3・5.4** および **6.4**）。 |

**補足**: マスタ読み取りは同時アクセス・API クォータを考慮し、キャッシュや再検知間隔を設計で定める。

## 4. 用語定義

| 用語 | 定義 |
|------|------|
| マスタ | スタッフ1人につき、氏名・部署・あだ名等を保持するデータの集合 |
| 反映 | 登録フォーム等で追加されたスタッフが、2つのアンケートで「選択可能な対象」として利用可能になること |
| 削除 | 退職フォーム等で処理されたスタッフが、2つのアンケートの選択対象から除外されること（物理削除か論理削除かは実装方針で決定） |
| 管理者 | **集計機能**（**5.15**）等、運用上 **特権を付与されたアカウント**。付与方法は設計で定める（ロール・メールホワイトリスト等）。 |
| マイ回答 | ログインユーザ本人が送信したアンケート回答のこと。**マイ回答・履歴**（**5.12**）で閲覧する。 |

## 5. 機能要件

### 5.1 アンケートフォーム（ソレイイネ!! 入力フォーム）

| ID | 要件 |
|----|------|
| F-ANK-01 | サイト上で「ソレイイネ!! 入力フォーム」を回答できること。 |
| F-ANK-02 | 回答対象のスタッフは **マスタに登録され、退職処理されていない** 者に限定すること。 |
| F-ANK-03 | 対象者の選び方として、**部署ごとの一覧** から名前を探せること（現行Googleフォームのプルダウンで回答者を選ぶ操作に相当）。 |
| F-ANK-04 | **名前検索** により対象者を絞り込めること。**あだ名** でもヒットすること。 |
| F-ANK-05 | 画面に表示する **タイトル・説明文・設問ラベル・プレースホルダ** は、現行Googleフォーム（**5.8**）と **同一文言** とすること。 |
| F-ANK-06 | 設問はいずれも **必須** とすること（現行フォームの必須指定に合わせる）。 |

### 5.2 アンケートフォーム（MVBe）

| ID | 要件 |
|----|------|
| F-MVB-01 | サイト上で「MVBe」用のアンケートを回答できること。 |
| F-MVB-02 | 回答対象のスタッフは **マスタに登録され、退職処理されていない** 者に限定すること。 |
| F-MVB-03 | **部署ごとの一覧** から名前を探せること。 |
| F-MVB-04 | **名前検索** により対象者を絞り込めること。**あだ名** でもヒットすること。 |
| F-MVB-05 | 画面に表示する **タイトル・説明文・設問ラベル・プレースホルダ** は、現行Googleフォーム（**5.9**）と **同一文言** とすること。 |
| F-MVB-06 | 設問はいずれも **必須** とすること（現行フォームの必須指定に合わせる）。 |
| F-MVB-07 | 説明文のとおり **幹部陣はMVBeの選出対象に含めない** こと（マスタ上で識別し、選択肢から除外する）。 |
| F-MVB-08 | **回答者** は現行フォームでは短文自由記述だが、新サイトでは **マスタからの選択**（部署別一覧・名前検索）とし、ラベルは現行どおり **「回答者」** とする。 |

### 5.3 スタッフ登録フォーム（Googleフォーム）回答者の自動反映

| ID | 要件 |
|----|------|
| F-REG-01 | スタッフ登録は **Googleフォーム** で受け付ける（現行運用との整合または移行方針に合わせる）。 |
| F-REG-02 | 登録フォームの回答が確定したタイミングで、マスタに **新規スタッフが登録** されること。 |
| F-REG-03 | 登録されたスタッフは、**5.1・5.2 の両アンケート** で、選択可能な対象として **自動的に利用可能** になること（手作業での二重入力を不要にする）。 |

※マスタの所在は **Google スプレッドシート** とする（**2.3**）。Googleフォーム→マスタ更新の技術手段（**Google Apps Script** によるシート直接更新、**Webhook 経由でサイトの API がシート更新** 等）は設計フェーズで選定する。中間DBのみを正とする二重管理は避ける。

### 5.4 退職フォーム回答者の自動削除

| ID | 要件 |
|----|------|
| F-RET-01 | 退職は **フォーム（Googleフォームを想定しうる）** で受け付ける。 |
| F-RET-02 | 退職フォームの回答に基づき、該当スタッフがマスタ上で **アンケート選択対象から除外** されること（「自動削除」の解釈に合わせ、論理削除を推奨候補とする）。 |
| F-RET-03 | 除外後、**5.1・5.2 の両アンケート** で当該者を選べないこと。 |

※過去回答データの保持ポリシー（匿名化・集計のみ残す等）は別途定義する。

### 5.5 部署ごとの分け方（一覧・選択UI）

| ID | 要件 |
|----|------|
| F-DEP-01 | マスタに **部署** が保持されていること。 |
| F-DEP-02 | アンケート回答画面では、**部署単位** でスタッフ一覧を閲覧・選択できること（現行のプルダウンで部署や名前を辿る体験に近いこと）。 |
| F-DEP-03 | 部署の並び順・未分類の扱いは運用ルールとして定義できること（初期はアルファベット順／登録順等で可）。 |

### 5.6 名前検索（あだ名対応）

| ID | 要件 |
|----|------|
| F-SRC-01 | アンケート画面上に **検索入力** を設けること。 |
| F-SRC-02 | **氏名（正式名称）** の部分一致または前方一致等、検索仕様は実装時に定める（ユーザビリティを優先）。 |
| F-SRC-03 | **あだ名** フィールドの内容でも検索ヒットすること。 |
| F-SRC-04 | 検索結果は部署情報と併せて表示し、誤選択を減らすこと。 |

### 5.7 共通：5つのValue（Be）の定義文

両アンケートで参照する「5つのBe」の意味づけは、各フォームの説明・設問で用いる文言として、次のとおりとする（**ソレイイネ!! 入力フォーム** と **MVBe** で表記が異なる箇所は、それぞれ **5.8** / **5.9** に記載した文を正とする）。

| Value | キャッチコピー（表記の骨子） |
|-------|------------------------------|
| Be better | よりよくあれ |
| Be honest | 正直であれ |
| Be proactive | 主体的であれ |
| Be challenging | 挑戦者であれ |
| Be authentic | 本物であれ |

### 5.10 UI方針（案）

本節は **画面の方向性のたたき台** とする。ピクセル単位の確定や Levela ロゴ等のビジュアルアセットの最終仕様は **2.2** のとおり本書の対象外とし、画面遷移図・コンポーネント設計で具体化する。**未認証時** の **ログイン画面** と、認証後の **ダッシュボード型シェル** の両方を定義する。後者は **左サイドバー＋上ヘッダー＋中央ワークスペース** の3区分で構成する。見た目・情報密度の参考として、**プロジェクトで共有する業務用ダッシュボードのスクリーンショット**（濃色のグローバルナビ、ライトな作業領域、ワークスペース上部のタイトル帯と検索・ツールバー、階層付き一覧の開閉操作に相当するUI）に準じた **プロフェッショナルで落ち着いた業務ツール** の方向性を取る。

**ダッシュボード型シェル（全体レイアウト）**

- **3 区分**: **左：グローバルナビ（サイドバー）**、**上：アプリヘッダー**、**中央：メインコンテンツ（ワークスペース）** とする。
- **サイドバー**: 幅は実装で定める（ブレークポイントに応じた折りたたみ可）。背景は **濃色（ネイビー／スレート系）** とし、**アイコン＋ラベル** の縦並びメニューとする。メニューは **グループ見出し**（例：メイン／その他）で整理できるようにする。現在ページは **ハイライト** 等で明示する。
- **ヘッダー**: サイドバーと調和した **濃色** を基本とする。左に **サービス名・ロゴ領域**（**2.2** の最終アセットは別途）、右に **ヘルプ・設定・アカウント／プロフィール** 等の配置余地を確保する（認証・表示内容は **8章** で確定するまでプレースホルダ可）。
- **メインコンテンツ（ワークスペース）**: **白〜ライトグレー** を基調とし、フォームや一覧を **カード／パネル** 上に配置する。表形式・リストにおいて **行区切り・ホバー・選択状態** が判別しやすいスタイルを許容する。
- **ワークスペース上部（サブヘッダー／ツールバー）**: 参考ダッシュボードと同様、**左に当該画面タイトルとアイコン**、**右に検索入力・補助操作** を配置できる **帯** とする。アンケート画面では **スタッフ名検索**（**F-SRC**）を本帯に置くか、スタッフピッカー直上に置くかは実装で決め、**視認性・一貫性** を優先する。
- **階層・開閉**: 参考UIの **ツリー一覧（すべて開く／折りたたみ、階層インデント）** に相当する操作感として、部署別スタッフ一覧は **アコーディオン** または **折りたたみ可能なツリー** で表現する（**F-DEP**）。
- **環境切替（テスト／本番）のようなセグメントコントロール** は、参考画面ではワークスペースに存在する。**本サービスでは** 一般スタッフ向けに必須としない。Preview／本番の URL・Vercel 環境・運用で分離し **画面上の切替を省略** してよい。**運用者のみ** に環境切替 UI を出す等が必要な場合は設計で定める。
- **行単位の副次操作**（行末のメニュー等）が将来必要になる画面は、同一デザインシステムで定義する。初版のアンケート必須画面に **必須ではない**。

**ログイン画面（未認証時・UI）**

- **位置づけ**: **未認証** の利用者が最初にたどり着く **専用の全画面ページ** とする（**5.11**）。認証に成功したあと **ダッシュボード（ホーム等）** へ遷移する。**ログイン画面ではダッシュボードのサイドバー・業務ヘッダーは表示しない**（認証後シェルとの切り分け）。
- **レイアウト**: 画面全体を **淡色**（白〜ライトグレー、またはごく薄いグラデーション）の背景とし、**視線を中央に集める** 構成とする。ワイド画面では **中央寄せ** の **カード（パネル）** にログイン操作を集約し、**最大幅** を設けて可読性を確保する。狭い画面ではカードを **幅いっぱいに近づけ**（適度な余白は維持）、**縦スクロール** で完結する。
- **カード内の構成（上からの想定）**:
  - **ロゴ・サービス名**（**2.2** の最終ロゴは別途。プレースホルダ可）。
  - **短い説明文**（例：スタッフ向け統合フォームサイトである旨。文言は運用で確定）。
  - **ログイン操作**: 採用する認証方式に応じて **IdP ボタン**（「Google でログイン」「SSO でログイン」等）、または **メール／パスワード** 入力欄＋**ログイン** ボタン等（**8章**・設計で確定）。**プライマリボタンは1つ** に絞り、**アクセントカラー** はログイン後のダッシュボードと **同一系統** とする（**ビジュアル・トーン** と整合）。
  - **補助リンク**（任意）: ログインできない場合の **問い合わせ先** や **社内手順** への導線。パスワード再設定を自サイトで扱う場合は **リンク領域** を確保する（実装方式は設計で決定）。
- **状態表示**:
  - **エラー**: 認証失敗・入力不備時は、カード内の **フォーム直上またはサマリ** に **簡潔なメッセージ** を表示する（技術的詳細の列挙は避ける）。
  - **処理中**: 二重送信防止のため **ログインボタンを無効化** し、**スピナー** または **ローディング表示** を付与する。
- **アクセシビリティ**: ラベルと入力の関連付け、キーボード操作、コントラストは **WCAG を意識** し設計で検証する（レベル目標はプロジェクトで定める）。
- **モバイル**: **タップ領域** を十分に確保し、仮想キーボード表示時も主要操作が隠れないよう余白を取る。
- **補足**: ログイン画面の **文言・項目の最終リスト** は認証方式確定後に定義する。ピクセル値・フォントサイズの固定は **2.2** の対象外とする。

**サイト全体・ナビゲーション（サイドバー中心）**

- **未認証時** は **ログイン画面**（上記）を経由し、認証後に **同一ダッシュボード内のナビゲーション** を利用する。**入口の一本化** は、ログイン後の **サイドバー** で実現する。**ログアウト・プロフィール** は **ヘッダー右側のアカウント領域** に置き、サイドバーには画面遷移用の項目を並べる（**5.11**）。
- **サイドバーに並べる項目の想定**（グループ見出しは実装で調整可）:
  - **メイン**: **サイトトップ（ホーム）**、**ソレイイネ!! 入力フォーム**、**【Levela】MVBeアンケート**。
  - **自分の記録**: **マイ回答・履歴**（**5.12**）。**回答状況**（今週・今月。**5.13**）は専用のサイド項目とするか、**ホーム上のサマリのみ** に集約するかは実装で選択する（いずれかまたは併用で要件を満たすこと）。
  - **共有**: **月間ランキング**（過去月切替。**5.14**）。
  - **管理**（**管理者** のみ表示。**5.15**）: **集計**。
  - **その他**: **スタッフ登録・退職**（Google フォームへの外部リンク）、**ヘルプ** 等（任意）。
- **サイトトップ** では案内に加え、**今週・今月の回答状況**（**5.13**）のサマリや、主要2フォームへの **ショートカット** を併置してもよい（**主ナビはサイドバー** とする）。
- **スタッフ登録・退職** 用の Google フォームへのリンクは、サイドバーの **「その他」グループ**、または **ワークスペース下部のフッター** 等に置き、本サイトが「回答専用」であることが伝わる表現とする。

**スタッフ選択UI（共通）**

- プルダウンでの大量選択は避け、**F-DEP・F-SRC** に沿い **検索を前面** に置く。
- **画面上部（またはピッカー内の先頭）に「名前・あだ名で検索」** を設け、入力に応じて候補を絞り込む。各候補は **氏名・部署・あだ名** を1行で示し、誤選択を減らす（F-SRC-04）。
- **部署単位の閲覧** として、**アコーディオン**（部署名をタップで展開し、その中にメンバー一覧）を想定する。PC では **左：部署一覧／右：メンバー** の2ペインも選択肢とする。部署内の並びは **F-DEP-03** の運用ルールに従う。
- **選択の確定** は、原則 **1タップで確定** とし、フォーム上に **「選択中：〇〇（△△部）」** 等の表示で状態を明示する。確認モーダルは実装時に運用負荷と照らして任意とする。
- 上記ピッカーは **1コンポーネント** として実装し、ソレイイネ!!（回答者名・賞賛対象）、MVBe（回答者・各 Be で選ぶ相手）で **同一の操作感** に揃える。

**ソレイイネ!! 入力フォーム画面**

- **縦1カラム** のフォーム流れとし、現行 Google フォームに近い体験とする。
- 冒頭に **5.8** どおりタイトル・説明文全文を表示する。5つの Be の説明は **読みやすい余白・見出し階層** を確保する。
- 設問順は **回答者名 → 賞賛したい人 → Value → 具体的内容**。Value は **ラジオボタンまたはセグメントボタン** 等で5択とし、長いラベルは折り返し前提の幅を確保する。
- フッターに **送信** と **フォームをクリア** を配置する。クリア時の **二段階確認**（確認ダイアログ）は実装時に検討する。

**MVBe アンケート画面**

- **5.9** どおり、説明文ののち **回答者** を共通ピッカーで選択する。
- **5つの Be ブロック** は、**カード** または **番号付きセクション**（例：1/5 … 5/5）で視覚的に区切り、長いフォームでも位置が分かるようにする。各ブロックは見出し（`■Be …`）→ 説明文 → メンバー選択 → 理由の短文の順とする。
- **進捗表示**（例：未入力ブロック数、完了ブロック数）を付与すると、離脱・送信前の不安を軽減しやすい。
- **幹部陣の除外**（F-MVB-07）は **候補リストに表示しない** ことで満たし、UI 上に追加の注意書きを過剰に増やさない（説明文の注意で補足済みとする）。

**ビジュアル・トーン**

- **ナビゲーションは濃色、作業領域は淡色** とする **高コントラスト** を基本とし、添付イメージに近い **プロフェッショナルでミニマルな業務ダッシュボード** の見え方を目指す。
- 説明文に絵文字等が含まれるため、ワークスペース内の本文は **堅すぎないが業務ツールとして信頼できる** トーンとする。
- **タイポグラフィの階層**（ページ見出し／セクション説明／設問ラベル／入力欄）を明確にし、長文説明も読みやすくする。
- **アクセントカラー** は送信・フォーカス・選択中表示・サイドバー選択中などに **1系統** に絞り、2フォーム間で統一する。
- ロゴ等は **2.2** のとおり最終仕様は別途とし、ヘッダー左に **プレースホルダ領域** を確保できるようにする。

**モバイル・検証・エラー表示**

- スマートフォン等の狭い画面では、サイドバーを **ドロワー／オーバーレイ** に格納し、**ハンバーガー等で開閉** するパターンを想定する。
- スタッフピッカーは **全画面ボトムシート** 等で検索と一覧の切り替えがしやすい構成を検討する。
- 送信時の必須エラーは **該当フィールド付近** に表示する。MVBe のように縦に長い画面では、**最初の未入力箇所へスクロール** する挙動があるとよい。

**補足（ログインとフォームの連携）**

- **5.11** のログインとマスタを突合し、**「回答者」を自動入力または省略** できる余地がある。認証プロトコルの詳細は **8章** で確定する。

## 6. データ要件

### 6.1 管理場所

スタッフの **氏名・部署名・あだ名** 等の **正のデータ** は、**当面、Google スプレッドシート上のマスタ用シート** で管理する（**2.3**）。将来的にデータベースへ移行する場合も、移行期間中の二重正を作らない運用を設計で定める。

| 方式 | 概要 |
|------|------|
| A. データベース | 正規化・検索・同時更新に有利。将来の拡張候補。 |
| B. スプレッドシート | **初期採用**。運用担当が直接編集しやすい。Googleフォーム連携と親和性が高い。 |

**要件**: 登録・退職の自動連携が成立すること。最終的な **正のデータ** が **マスタ用シートに一意に** 明確であること（二重管理による不整合を防ぐ）。

### 6.2 マスタに想定する項目（最低限）

| 項目 | 必須 | 備考 |
|------|------|------|
| スタッフID | ○ | システム内部識別子（連携・重複防止） |
| 氏名（表示名） | ○ | アンケート一覧表示用 |
| 部署 | ○ | グルーピング・表示用 |
| あだ名 | △ | 検索用。未設定可とするか運用で決定 |
| 幹部フラグ（等） | △ | **MVBe** で選出不可の「幹部陣」を判定し、選択肢から除外する（**5.9**・F-MVB-07）。運用で名称・階層を定義する。 |
| 在籍状態 | ○ | 在籍／退職等。アンケート対象判定に使用 |
| 登録日時・更新日時 | 推奨 | 監査・トラブルシュート用 |

追加項目（メール、社員番号、読み仮名等）は運用・連携要件に応じて拡張する。

### 6.3 スプレッドシート構成（1ブック・複数シート）

1つの Google スプレッドシート（ブック）に、少なくとも次の **シート（タブ）** を設ける想定とする。シート名は実装・運用で確定してよいが、役割は次のとおりとする。

| シート（例） | 用途 |
|--------------|------|
| `master`（例） | **6.2** のマスタ項目を保持する。**正のマスタ** とする。 |
| `responses_soreine`（例） | 「ソレイイネ!! 入力フォーム」相当の **サイト上送信** 1件につき1行（送信日時、回答者・賞賛対象の識別子または表示名、選択した Value、自由記述テキスト等）。列の詳細は設計で定める。 |
| `responses_mvbe`（例） | 「MVBe」相当の **サイト上送信** 1件につき1行（送信日時、回答者、各 Be ブロックで選んだ相手・理由等）。列の詳細は設計で定める。 |

**補足**: 開発用（Preview）と本番用で **別ブック** を用意し、誤って本番データを書き換えない運用とする（Vercel の環境ごとにスプレッドシート ID を分ける等）。

### 6.4 サイトから回答シートへの書き込み

Next.js（Vercel 上）から **Google Sheets API** 等を用い、**回答シートへ行を追記**する。送信内容の妥当性（必須項目、在籍者のみ、MVBe の幹部除外等）は **サーバー側で再検証** する。

| 区分 | 要件の方向性 |
|------|----------------|

| 保守性 | 設問変更や部署再編にある程度追従できる構成とする。 |

## 8. 制約・前提（要確認事項）

1. **Googleフォーム継続の範囲**: **登録・退職は Google フォームのまま** とし、**2つのアンケート（ソレイイネ!!・MVBe）はサイト内フォームへ移行** する方針で進める。段階移行が必要な場合はリリース計画で定める。
2. **認証**: **ログイン機能** を提供する（**5.11**）。**プロトコル・IdP**（Google、社内SSO 等）は **2.3 の技術スタック上で実装可能な範囲で設計フェーズで確定**する。**管理者** の付与方法（マスタ列、IdP グループ、環境設定等）は **5.15** と合わせて設計で確定する。
3. **「自動反映」の遅延許容**: マスタの更新がサイトのスタッフ一覧に反映されるまでの許容時間（即時〜数分〜日次バッチ等）。**マスタ読み取りのキャッシュ TTL**（**2.3** 補足）と整合させる。
4. **退職の識別子**: フォームでは氏名のみか、**スタッフID** 等の一意キーを必須にするか（同名対策）。
5. **既存Googleフォーム回答データ**: 移行の要否、保管期間。新 `responses_*` シートは **サイト開設以降の回答のみ** とするか等。
